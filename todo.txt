todo gemini-found issues(may not be a thing):

* **Unreachable Buffer Growth:** In `QueryFullProcessName`, if the retry loop doubles the `nextSize` allocation to exactly `MaxExtendedPath` (32767), the subsequent boundary check `if nextSize > MaxExtendedPath` triggers an error, preventing the allocation from ever succeeding at the absolute limit.
* **Brittle UUID Generation:** `generateUniqueRuleID` hard-panics the server if it encounters 10 UUID prefix collisions. Crashing the proxy over an RNG collision during a whitelist addition is unnecessarily destructive.
* **Aggressive Stdin Flushing:** `ClearStdinIfTermIsNOTRaw` relies on `GetNumberOfConsoleInputEvents`. This Win32 API counts *all* input events—including mouse movements, window resizes, and focus events. It will aggressively flush the buffer even if the user never touched the keyboard.
* **Screen Reader Race Condition:** The WebUI blocks page sets `aria-live="polite"` on the `.block-action-feedback` span, but immediately overwrites its text content and class name upon receiving the network response. Screen readers frequently drop rapid, sequential DOM mutations on polite regions if they occur too closely together.
* **DNSSEC Validation Breakage:** In `processRR`, the code unconditionally filters out and drops `*dns.RRSIG` records (`return false, nil, StrippedRRSIG`). If a client requests DNSSEC validation (DO bit = 1) and the upstream provides the signatures, stripping them guarantees the client's validation will fail, causing it to reject the legitimate response as bogus.
* **Partial Batch Commits in WebUI:** In `applyTablesHandler`, if an error occurs midway through processing a batch of changes, the loop breaks and returns an `http.Error` to the client. However, it still calls `flushDirty()`, committing all the successfully processed changes *prior* to the error to disk. The client receives an error (assuming the batch failed), while the server actually applied half of it, causing state desynchronization.
* **Local Host IPv4/IPv6 Blackholing:** In `handleDNSQuery`'s local host override section, if a domain has only IPv6 addresses configured in `/hosts`, an `A` (IPv4) query will match the host but fail the `isIPv4` check. It then drops to the bottom of the block and returns a `NOERROR` response with 0 answers (NODATA). This prevents the query from falling through to the upstream resolver, effectively blackholing IPv4 clients for that domain instead of resolving the upstream IPv4 address.
* **Missing HTTP Cache-Control on DoH Responses:** In `dohHandler`, GET requests containing the `dns` query parameter are successfully parsed and served, but the HTTP response lacks a `Cache-Control` header derived from the DNS response's TTL. Intermediate proxies or browsers might cache the HTTP response indefinitely or incorrectly, serving stale DNS records to DoH clients.
* **Missing TCP Keep-Alives on DNS Server:** In `runDNSTCPLoop`, incoming TCP connections accepted via `tcpLn.Accept()` are never configured with TCP Keep-Alives (`SetKeepAlive(true)`). Idle, broken, or misbehaving client connections could linger silently in the OS network stack until the application-layer read/write deadlines finally expire.
* **Uncancellable Goroutine Leak in Metadata Lookup:** The `startMetadataLookup` function spawns a `s.GoSafe` goroutine to execute `wincoe.PidAndExeForUDP` or `PidAndExeForTCP`. These underlying Win32 lookup functions do not accept a `context.Context`. If the OS-level system calls hang (e.g., due to a locked WMI or exhausted system handles), the goroutine will block forever and leak, even though the parent request context times out.
* **Incomplete 0x20 Case Randomization Support:** The `adjustResponseCaseToQuery` function attempts to restore the client's original casing to the response to support 0x20 strict clients. However, it only adjusts records where `hdr.Name` matches the query name exactly. If the upstream returns a `CNAME` chain, the intermediate `CNAME` targets will not have their casing adjusted, which can still cause strict resolvers to reject the chain.
* **In-Flight Log Drops During Reload:** In `LoggerManager.Reinit`, the global logger pointer is swapped, and then `Close()` is called on the old log writers. If an active DNS query goroutine grabbed the old logger pointer a microsecond before the swap, it will attempt to write to it *after* `Close()` has executed. The `asyncLogWriter` safely catches this via its `w.closed` flag, but it explicitly drops the log line instead of migrating it to the new logger.
* **Blind Type Assertion Panic in Cache:** In `goCacheStore.Get`, if the `go-cache` instance somehow stores an item that isn't of type `CacheEntry`, the `else` block executes `panic2("BUG: not of CacheEntry type")`. A shared underlying cache map corruption would instantly bring down the entire server on the next read.
* **Config Rollback Race Condition:** In `UpstreamManager.updateInnerState()`, the code creates a clone of the live config, computes new upstream fields, and writes the clone back via `um.liveConfigs.Store()`. If a WebUI user successfully applies a configuration change (which triggers `s.applyConfig()`) at the exact same moment a network failure triggers `buildSet()`, the `updateInnerState` routine will overwrite the user's fresh configuration with its older, cloned version, silently reverting their changes.
* **Terminal Raw-Mode State Corruption:** In `WaitAnyKeyIfInteractive()`, `WithConsoleEventRaw()` is executed simultaneously on the main thread and inside a spawned goroutine. Because both routines fetch the current terminal mode via `GetConsoleMode`, alter it, and restore it via `defer`, an overlap guarantees that one routine will capture the *already-modified* raw state as its "original" state. Upon exit, the user's terminal will be left permanently broken in raw mode.
* **Rate-Limiter Fast-Path Blocking:** In `handleDNSQuery`, if a query exceeds the rate limit, the handler blocks on `future.wg.Wait()` to extract the executable name for the warning log before dropping the request. Because the future resolves via synchronous OS-level WMI/TCP table checks, a UDP flood will tie up thousands of goroutines waiting for the OS, entirely defeating the rate limiter's purpose and instantly exhausting system resources.
* **Unbounded Login Tracker Growth:** The `LoginTracker.records` map allocates a new `loginRecord` for every unique IP that fails authentication (`RecordFailure`). Because there is no background janitor or size cap for this map, an attacker sending malformed basic-auth requests with spoofed IPs (or via a botnet) will grow this map infinitely until the application crashes from an Out-Of-Memory (OOM) exception.
* **Missing Signal Catch During Boot:** In `OldMain()`, `signal.Notify` is not registered until execution reaches `Server.Run()`. If an administrator presses `Ctrl+C` during the potentially lengthy `LoadAndValidateConfig` or `initAdminUI` phases, the default Go runtime signal handler takes over and terminates the process instantly. The deferred `rt.FlushLogsForShutdown()` is bypassed, meaning critical startup diagnostic logs are permanently lost in the buffer.
* **O(N) Synchronous Cache Eviction:** `invalidateCacheForPattern` and `invalidateCacheForBlacklistedIPs` iterate synchronously over the entire `cachee.Items()` map. If the WebUI submits a batch of 100 host/rule modifications via `applyTablesHandler`, the server executes 100 consecutive full-table scans on the HTTP handler thread, causing severe CPU spikes and blocking the WebUI response.
* **Punycode Expansion Length Violation:** In `punycodeEncodePattern`, Unicode labels are converted to ASCII `xn--` formats via `idna.Lookup.ToASCII()`. However, the code never checks if the expanded punycode label exceeds the strict DNS 63-character limit or the 253-character total limit. These silently oversized rules are saved to disk but will inherently fail to match any valid wire-format DNS query.
* **Inconsistent Wildcard Validation:** `punycodeEncodePattern` explicitly throws an error if wildcard tokens are mixed with Unicode characters in the same label (e.g., `café*.com`). However, purely ASCII patterns skip this function entirely via the `isASCII(pattern)` fast-path. This creates an arbitrary validation inconsistency where `test*.com` is successfully saved, but `café*.com` is rejected.
* **Bracketed IPv6 Parsing Fallback:** In `getCleanIP()`, if `net.SplitHostPort` fails on an un-ported, bracketed IPv6 address (e.g., `[fe80::1]`), the code falls back to parsing the raw string. `net.ParseIP` inherently fails on bracketed IPs, returning `nil`. This causes the function to bypass the loopback collapsing logic (`parsed.IsLoopback()`), returning the raw bracketed string and fracturing the rate-limiter's IP tracking.
* **Invalid Content-Length on DoH Errors:** In `dohHandler`, if the DNS response packing fails, the code executes `w.Header().Set("Content-Length", ...)`, followed by `w.WriteHeader(http.StatusOK)`, and then attempts `w.Write(pack)`. If the write fails partially or fully, the HTTP response goes out with a `200 OK` status and a fixed `Content-Length` that does not match the actual truncated payload, causing strict HTTP clients to hang while waiting for the remaining bytes.
* **Unbounded Upstream Payload Read (OOM Vector):** In `doSingleDoHRequest`, if the upstream HTTP response returns a `200 OK`, the proxy reads the response via `io.ReadAll(resp.Body)`. While the incoming client request is heavily guarded by `DoHMaxRequestBodyBytes`, there is zero size limit enforced on the *upstream's* response. A compromised or misconfigured upstream returning a multi-gigabyte payload will cause `io.ReadAll` to consume all available server RAM, leading to an immediate Out-Of-Memory (OOM) crash.
* **Strict Mode Semaphore Exhaustion (Deadlock):** In `ForwardToDoH`'s strict mode, `wg.Wait()` blocks until every single upstream goroutine finishes its HTTP request. If the client drops the connection or the 5-second incoming context times out, the parent goroutine remains blocked at `wg.Wait()` until the upstreams hit their internal timeouts. Because this parent goroutine holds a precious slot in `dnsUDPSem` / `dnsTCPSem`, a slow upstream combined with a burst of client queries will instantly exhaust the semaphore pool, locking up the entire server and preventing any new connections from being accepted.
* **Cache Miss Thundering Herd:** In `handleDNSQuery`, if a highly queried domain expires from the cache, every concurrent incoming packet for that domain will experience a cache miss simultaneously. The server will spawn a completely independent upstream request for *every single packet*. Without a request-coalescing mechanism (like singleflight), a brief burst of 500 UDP queries for the same expired domain will result in 500 duplicate HTTPS requests hammering the upstream provider, risking rate-limits or bans from services like Quad9 or Cloudflare.
* **Blind Trust of Bcrypt Formatting (WebUI Lockout):** In `configHandler`, when a user updates their password, the code checks if the input is already a hash via `isValidBcryptHash(plainPwd)`. If an attacker or script submits a dummy string that perfectly matches the 60-character bcrypt format (e.g., exactly 60 characters starting with `$2a$10$`), the server blindly accepts it as a pre-hashed password and persists it to disk. Since it is a dummy string, the plaintext is unknown, permanently locking all administrators out of the WebUI.
* **Punycode Prefix Decoding Failure:** In `punycodeDecodePatternForDisplay`, the code attempts to convert `xn--` labels back to Unicode using `idna.Punycode.ToUnicode(label)`. However, Go's `Punycode.ToUnicode` strictly expects the string *without* the `xn--` prefix. Because the code passes the raw label (which still contains the prefix), the decoder will fail or return garbage. IDN domains will permanently fail to decode for display, leaving them unreadable in the WebUI and logs.
* **Unclamped UDP Response Transmission:** In `handleUDP`, the proxy extracts the client's advertised UDP size from the EDNS0 OPT record (`clientOpt.UDPSize()`) and passes it directly to `resp.Truncate(maxUDPSize)`. If a client maliciously or erroneously advertises a maximum size of 65,535 bytes, the proxy will attempt to pack and transmit a UDP payload of that exact size. It completely fails to clamp the client's request against the server's own `cfg.DNSUDPBufferSize` (default 4096), resulting in transmission failures or severe network fragmentation.
* **Stalling Log Writer Deadlock:** In `rotatingLogWriter.Write`, if a disk stall is detected, the code writes a warning to `os.Stderr` and then immediately attempts to write a secondary warning directly into the log file (`w.file.WriteString(warningMsg)`). Because it does this while still holding `w.mu.Lock()`, if the disk is genuinely hung, this secondary write will block indefinitely. Every other goroutine attempting to log will queue up, fill the `asyncLogWriter` channels, and subsequently drop all incoming log data.
* **Insecure Staging File Creation (Symlink Vulnerability):** In `GenericSafeFileWriter.SafeWriteFile`, the staging file (e.g., `config.json.powergotlost`) is created using `os.OpenFile(..., os.O_CREATE|os.O_TRUNC, perm)`. It crucially omits the `os.O_EXCL` flag. If a malicious script running as a lower-privileged user pre-creates the `.powergotlost` file as a symlink or hardlink pointing to a sensitive system file, the proxy will follow the link and overwrite the protected file with DNS configuration data.
* **Pointer Arithmetic Out-of-Bounds in SCM Parsing:** In `GetServiceNamesFromPIDUncached`, the code checks if the `ServiceName` pointer is within the bounds of the returned buffer: `if snPtr < bufStart || snPtr >= bufEnd`. However, strings in C/Win32 are null-terminated. This logic only verifies that the *start* address of the string is inside the buffer. A corrupted or malformed Service Control Manager response where the pointer is exactly at `bufEnd - 1` will cause `UTF16PtrToString` to read past the end of the allocated buffer into unmapped memory, triggering a hard segfault.
* **CSRF Token Fixation:** In `csrfMiddleware`, if a browser sends a `csrf_token` cookie, the middleware extracts the value (`token = cookie.Value`) and blindly injects it into the request context. There is no server-side validation ensuring that the token was cryptographically signed or originally issued by this server. An attacker capable of setting a cookie on the victim's browser (e.g., via a subdomain XSS) can fixate the token to a known value, completely bypassing the CSRF protection.
* **Windows Registry Handle Leaks:** In routines that interact with the Windows OS (e.g., reading network adapter configurations or service installation), if `registry.OpenKey` is used, the returned `registry.Key` handle must be explicitly closed. If the code encounters a mid-function error (e.g., a missing subkey) and returns early without calling `k.Close()` or deferring it, the proxy will silently leak OS handles. Over time, this exhausts the system's handle quota, causing downstream OS operations to fail.
* **32-bit Architecture Alignment Panics:** If 64-bit atomic counters (such as those tracking total bytes processed or total queries dropped) are placed arbitrarily inside large structs like `Server` or `UpstreamManager`, they may not be 64-bit aligned in memory. While this runs perfectly on 64-bit Windows, compiling and running the proxy on a 32-bit Windows system (x86 or ARM32) will result in a hard runtime panic the moment those atomic variables are accessed.
* **Deferred Body Close on Edge-Case HTTP Errors:** In the DoH client implementation, a call to `http.Client.Do(req)` can occasionally return a non-nil `error` *alongside* a non-nil `Response` (most commonly during custom redirect policy failures or specific HTTP/2 stream errors). If the error-checking block simply logs and returns without explicitly checking for a non-nil response to call `resp.Body.Close()`, the proxy will leak the underlying TCP connection and socket descriptor.
* **Transaction ID Pass-Through Vulnerability:** If the proxy supports forwarding traditional UDP port 53 traffic to non-DoH upstreams, simply passing the client's original DNS packet through without rewriting the Transaction ID is dangerous. It relies entirely on the client OS for entropy. If an IoT device on the network uses predictable or static Transaction IDs, an off-path attacker can easily flood the proxy with spoofed UDP responses, poisoning the proxy's global cache.
* **EDNS0 Client Subnet (ECS) Privacy Leak:** If the proxy caches DNS responses, it must heavily scrutinize upstream responses containing EDNS0 Client Subnet data. If a client queries a domain, and the upstream returns an IP specific to that client's subnet, caching that response globally means all subsequent clients (regardless of their subnet or privacy settings) will be served the IP optimized for the first client, potentially leaking location topology and breaking CDN routing.
* **Stored XSS via Malicious DNS Queries:** The WebUI renders live traffic on the `/logs` and `/logs_queries` pages. The DNS protocol technically allows almost any byte in a domain name label. If an infected machine on the local network blasts DNS queries crafted as HTML payloads (e.g., `<script>...</script>.com`), and the Go backend passes these to the frontend without strict contextual sanitization, viewing the logs will execute the payload in the administrator's browser context.
* **JSON Type Confusion Panics:** When the WebUI submits configuration changes via the `applyTablesHandler`, the backend likely parses the JSON payload. If it unmarshals this payload into a generic structure and performs blind type assertions (e.g., assuming a `timeout` field is always a `float64` or a `hostname` is always a `string`), a malformed JSON payload containing an array or object in those fields will trigger an unhandled runtime panic, killing the HTTP server thread.
* **Context Leak in Reload Cycles:** If background watchers or polling routines (like `Server.watchKeys`) are instantiated using a new context during a configuration reload, the code must guarantee the `cancel()` function of the *previous* context is called. If the proxy relies on garbage collection to clean up old contexts instead of explicit cancellation, the internal channel structures and potentially blocked goroutines tied to those old contexts will pile up in memory with every config save.

On Windows, `O_TRUNC` on an existing file preserves the existing file's Security Descriptors (ACLs). If a low-privilege user previously created a dummy `key.pem`, the new private key inherits those weak permissions despite the `0600` argument in `OpenFile`. Keeping the ACLs is good/intended, however we should ensure the 0600 is set ie. setattr?
//* **RFC 2308 Violation (Negative Caching):** In `handleDNSQuery`, negative responses (like `NXDOMAIN`) are cached using a strictly hardcoded TTL (`cfg.CacheNegativeTTLSec`). It completely ignores the `SOA` record's `MINIMUM` field in the Authority section, violating DNS RFCs which require using the minimum of the SOA's TTL and the `MINIMUM` field for negative caching.
//* **Destructive DOM Highlighting:** The `applyTableFilter` logic in `app.js` dynamically attaches `<mark>` elements directly into the DOM tree of table rows. When the user clears the search, it reconstructs the text nodes. This constant DOM shredding on every keystroke causes unnecessary reflows and GC pressure.
